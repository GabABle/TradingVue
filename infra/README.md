# TradingVue infrastructure

Infrastructure-as-Code (Terraform) and CI/CD (GitHub Actions) for deploying
TradingVue to AWS in two isolated environments — **staging** and
**production** — where every change reaches staging first and production is
gated behind a manual approval.

## Architecture

```
                       ┌──────────────────────────── CloudFront (HTTPS) ───────────────────────────┐
   Browser ───────────▶│  default behaviour ─▶ S3 (static SPA)                                      │
                       │  /api/*            ─▶ ALB ─▶ ECS Fargate (Express API) ─▶ RDS PostgreSQL    │
                       └────────────────────────────────────────────────────────────────────────────┘
```

* **Frontend** (`artifacts/trading-chart`) — Vite SPA, built in CI, served from
  a private S3 bucket via CloudFront.
* **API** (`artifacts/api-server`) — Express, bundled to `dist/index.mjs`, run on
  ECS Fargate behind an internal ALB. CloudFront proxies `/api/*` to it, so the
  browser uses a single origin (no CORS).
* **Database** — RDS PostgreSQL in private subnets, reachable only from the API
  security group. Connection string lives in Secrets Manager.
* **Secrets** — `JWT_SECRET` is generated by Terraform; third-party API keys
  (Alpaca, OpenAI, Resend) start blank in Secrets Manager — fill them and redeploy.
* **Migrations** — `drizzle-kit push` runs as a one-off ECS task before each
  deploy (a separate `migrate` image target).

Each environment is a separate Terraform state and a fully separate set of AWS
resources. Staging is lean (1 task, db.t4g.micro, single-AZ); production is
sturdier (2 tasks, db.t4g.small, Multi-AZ, deletion protection).

## Layout

```
infra/
  bootstrap/            # one-time, account-level: TF state bucket, lock table,
                        # GitHub OIDC role, shared ECR repo
  modules/
    network/            # VPC, subnets, NAT, security groups
    database/           # RDS Postgres + DATABASE_URL secret
    api/                # ECS cluster, ALB, task defs (api + migrate), service
    frontend/           # S3 + CloudFront (SPA + /api proxy)
    stack/              # composes the above + publishes SSM deploy params
  environments/
    staging/            # terraform root for staging
    production/         # terraform root for production
```

## One-time bootstrap (run from AWS CloudShell)

```bash
git clone https://github.com/GabABle/TradingVue.git
cd TradingVue/infra/bootstrap

# Pick a globally-unique bucket name for Terraform state.
terraform init
terraform apply -var="state_bucket_name=tradingvue-tfstate-026818611950"

# Note the outputs — you'll need github_deploy_role_arn and state_bucket.
terraform output
```

This creates the state backend, the shared ECR repo, and the GitHub Actions
deploy role. It uses local state (kept in CloudShell); re-run only when these
account-level foundations change.

## Provision an environment

```bash
cd ../environments/staging
terraform init -backend-config="bucket=tradingvue-tfstate-026818611950"
terraform apply        # review the plan, then approve

# Repeat for production (only after staging is healthy):
cd ../production
terraform init -backend-config="bucket=tradingvue-tfstate-026818611950"
terraform apply
```

`terraform output app_url` prints the public HTTPS URL for the environment.

## Wire up GitHub Actions

In the repository (Settings):

1. **Variables → Actions** (repository variables):
   * `AWS_DEPLOY_ROLE_ARN` = the `github_deploy_role_arn` bootstrap output
   * `AWS_REGION` = `ap-southeast-1`
2. **Environments → New environment → `production`**: add a **Required
   reviewers** protection rule (yourself). This is the staging→production gate.
   Create a `staging` environment too (no protection needed).

After that, every push to `master`:

1. builds the API image + migration image + frontend once,
2. deploys to **staging** automatically (migrate → ECS → S3/CloudFront → smoke test),
3. **pauses for your approval**, then promotes the *same* artifacts to **production**.

## Filling in third-party API keys

```bash
# ARN is printed by `terraform output app_secret_arn`
aws secretsmanager put-secret-value \
  --secret-id tradingvue-staging/app \
  --secret-string '{"JWT_SECRET":"<keep-existing>","ALPACA_API_KEY_ID":"...","ALPACA_API_SECRET_KEY":"...","OPENAI_API_KEY":"...","RESEND_API_KEY":"..."}'
```

Then re-run the deploy (push, or re-run the workflow) so tasks pick up the new values.
The app boots without these — only the features that need them stay disabled.

## Changing infrastructure later

Edit the Terraform, then `terraform plan` / `apply` in `environments/staging`
first, and only afterwards in `environments/production` — same staging-first
discipline as application deploys.

## Teardown

`terraform destroy` in each environment directory. Production has
`deletion_protection = true` on the database; flip it to `false` and apply
before destroying.
